🚀 Release Notes: ButlerSDK v1.1.2-preview (Security Hotfix & Feature Update)
Note: This v1.1.2-preview is NOT on NuGet yet. When published, the ButlerSDK.Core package will be bumped to v1.1.2-preview as it holds these changes. Unchanged packages will remain at v1.1.1-preview.


Summary:
This update introduces a major security hardening of the ButlerTool_LocalFile_Load file-system tool, transitioning it from a standard path-filter to a bounded, OS-locked sandbox to expose only dev supplied locations.  It also introduces the new AttachFile() method to give developers safe, explicit control over out-of-sandbox file exposure. 
All developers utilizing LLM file-access tools are strongly encouraged to review these changes.


🛡️ The Sandbox: What it is and What changed 
Before v1.1.2:
The file tool essentially acted as a passthrough. If an LLM hallucinated a path, or a user used prompt injection to ask the LLM to read C:\Windows\System32\drivers\etc\hosts or ~/.bashrc, the tool would execute the read/write as long as the host application had the OS permissions to do so. The LLM had the run of your hard drive.


Now (v1.1.2+):
New Feature: ButlerTool_LocalFile_Load  now has a read sandbox, a write sandbox and the ability to attach file case by case basis to escape.


ButlerTool_LocalFile_Load operates on a strict Default-Deny architecture. The LLM is trapped by default.

How it works: You must explicitly mount safe folders to the tool using AddSandBoxPath(path, filter) (e.g., Read, Write, or Both). Skipping this will default to creating a blank temporary folder in your system temporary folder location.
The Intercept: When the LLM requests a file, the SDK intercepts the path, strips out traversal attempts (../../), follows all symlinks to their true physical location, and verifies that the final destination lives entirely inside your mounted folders.
The Result: If the LLM tries to touch anything outside the mounted sandbox, the SDK blocks the OS call entirely and simply feeds a text error back into the LLM's context window.


🏗 Architecture & API Updates
New Feature: Developer-Controlled AttachFile()
This update adds an AttachFile() method that allows a single file to be safely exposed to the LLM outside the normal sandbox directory. Symlinks are resolved to their real target (file‑only; directory symlinks are rejected).

This ensures:
1.The sandbox remains strict by default without it being cumbersome to white list. 
2. The LLM cannot easily expand its own access unless explicitly given a tool to attach files.
3. Developers can intentionally expose specific datasets (e.g., C:\Data\Dataset.txt).
4. Accidental “how did the model access this?” scenarios are drastically reduced.
5. It makes the sandbox exceptionally harder to break out of—no code is perfect, but we're no longer leaving the key in the front door.

The Why:
It became necessary (good dev experience) while addressing Issue #8 to make it easy for devs to attach files outside the sandbox (similar to uploading a file to ChatGPT/Gemini).
WARNING: If you create a custom tool that enables the LLM to attach its own files without a code-enforced boundary, you are potentially exposing your PC's entire drive to the LLM depending on prompt injection.


🔒 Security & Governance (what was bad, what was fixed)
Critical Vulnerability Patched (The "Wormhole" Directory Symlink Escape):
Impact if Unpatched: An attacker using prompt injection could trick the LLM into reading or writing through a directory symlink created inside the sandbox that actually points to a vital system directory (e.g., rewriting the Windows hosts file to redirect bank URLs to malware IPs). Standard string-based path checks would incorrectly validate the path as safe.
The Fix: Implemented a deep recursive FollowLink stack-resolver. The engine now verifies the true physical path of every parent directory in the chain, ensuring the final resolved target remains physically bounded within the sandbox restrictions.


Critical Vulnerability Patched (TOCTOU Race Conditions):
Impact if Unpatched: Time-Of-Check to Time-Of-Use (TOCTOU) allowed a microscopic race condition where an attacker with background host access could swap a validated "safe" file for a malicious symlink in the exact millisecond before the SDK executed the read/write operation.
The Fix: Replaced standard File.ReadAllText/WriteAllText with a "Lock and Verify" pattern. TOC_TOU_WriteText and TOC_TOU_ReadText now utilize FileStream with FileShare.None. The OS places a strict kernel-level lock on the file during operations, making it harder to swap out when acting on the file(s)


Vulnerability Patched (Legacy DOS Device Deadlocks/Bypasses):
Impact if Unpatched: LLMs instructed to read legacy Windows device names (CON, LPT0, PRN) could cause application thread hangs, or bypass path validation entirely due to parsing discrepancies between local Windows machines and CI/CD environments (like GitHub Actions).
The Fix: Added a strict, deterministic, cross-platform BlackListed routine. It surgically filters reserved DOS names without accidentally catching legitimate dictionary words (e.g., Computer.txt) or blocking developer dotfiles (e.g., .env).

Vulnerability Patched (OOM Memory Bombing):
Impact if Unpatched: An LLM requesting to load a massive file (e.g., a 10GB .log file) would cause the .NET runtime to attempt to allocate the entire file into RAM, throwing a fatal OutOfMemoryException and crashing the host application.
The Fix: Introduced FileReadSizeCap (defaulting to 20MB). Files exceeding this limit now safely throw a new, catchable FileIsGiganteException, which returns a clean error message to the LLM context window without crashing the pipeline. If you need larger size, subclass the tool as override FileReadSizeCap routine so it returns your new cap.
Hardened Pipeline (Fail-Closed Architecture):
The Fix: The core GetSecurePath routing is now wrapped in a strict fail-closed try/catch block. Unpredictable OS-level exceptions (such as infinite recursive symlink loops or PathTooLongException) now result in a graceful sandbox denial (null) rather than bubbling up and crashing the orchestrator.


If you can't upgrade:
Ensure you add some checking or keep the account running the LLM using ButlerSDK locked so it doesn't modify any files you don't want it too. I recommmand DO NOT RUN AS ADMIN or a Service account. Run it limited user if at all posble.

🚀 Release Notes: ButlerSDK v1.1.1-preview (Security Hotfix)

Summary:
This is a critical security patch addressing a Denial of Service (DoS) vulnerability discovered in the Tool Surface Scope validation engine introduced in v1.1.0-preview. All users utilizing VirtualTool architectures must upgrade to this version immediately.

The Window Vault Key Creator Helper CLI app will truncate the output file first to guard against data corruption.
 


🔒 Security & Governance
Critical Vulnerability Patched (VirtualTool Recursive DoS):
Fixed a severe vulnerability where the SDK performed an unchecked traversal of inner tools within a VirtualTool. Because the validation engine lacked state-tracking across nested layers, a VirtualTool that accidentally (or maliciously) referenced itself—or contained a circular dependency (e.g., Tool A -> Tool B -> Tool A)—would trap the engine in an infinite recursive loop.


Impact if Unpatched:
Resolving permissions for a cyclic VirtualTool triggers either a crash, likely StackOverflowException OR just will deadlock. In the .NET runtime, StackOverflowException is a fatal error that bypasses standard try/catch blocks and instantly terminates the entire host application process, acting as a highly effective Denial of Service (DoS) attack.


The Fix (Hardened Traversal Engine):
Cycle Detection: The ToolSurfaceFlagChecking engine now passes a HashSet<IButlerToolBaseInterface> (using ReferenceEqualityComparer) down the recursive chain. If a circular reference is detected, the engine safely ignores the duplicate and breaks the loop.
Strict Depth Limits: Introduced a hard cap on tool nesting (default: 2000 layers). This acts as a secondary failsafe to protect the CLR from stack depletion.
Catchable Exceptions: Exceeding the depth limit now safely throws a new, catchable VirtualTool_NestedOverflow security exception instead of crashing the process.


🏗 Architecture & API Updates
ToolSurfaceFlagChecking Refactor:
Introduced LookupToolFlagEx(...), which accepts the traversal HashSet and depth counters.
The standard LookupToolFlag routine now acts as a safe wrapper around LookupToolFlagEx, initializing the protective state tracking automatically.
Power-User Bypass: For edge cases requiring extreme depths, developers can toggle ToolSurfaceFlagChecking.AllowBigCheck to bypass the standard integer limits, shifting the responsibility to the host application.


⚠️ Mitigation Notes (If unable to update)
If you are locked to v1.1.0-preview and cannot immediately update, you must strictly audit your tool generation pipeline:
Ensure that any VirtualTool added to the Butler instance does not reference itself.
Manually verify that no deep circular references exist in your inner tool dictionaries.
Do not allow dynamically generated or untrusted third-party tools to be loaded into a VirtualTool without strict code review.



🚀 Release Notes: ButlerSDK v1.1.0-preview

Summary:
This release focuses heavily on Architectural Decoupling, Security Governance, and Provider Stability. It formally introduces the Tool Surface Scope security model and entirely decouples the Memory Engine from the Core Logic.

Note: Upgrading to v1.1.0-preview or higher is strongly recommended due to breaking enhancements in the ButlerSDK.Core.Abstraction data types and provider-specific storage objects. It's recommended to version match if at all reasonable possible.

🔒 Security & Governance

Capability-Based Security (ToolSurfaceScope): Tools are now recommanded to declare their operational intent via the [ToolSurfaceCapabilities] attribute (e.g., NetworkRead, DiskWrite, ArbitraryExecution).  

Active Firewalling: Butler.AddTool() now actively enforces security perimeters. If a tool requests permissions exceeding the Butler instance's allowed scope, the system immediately rejects it and throws a SecurityException.

Zero-Trust by Default: Untagged tools are safely assumed to require MaxPermissions and are blocked by default unless the hosting environment explicitly grants broad access.

Memory Hardening: Updated SecureStringHelper to support secure, character-by-character reading directly from Streams, minimizing memory exposure.

🏗 Architecture & Core Decoupling

Memory Engine Dependency Injection: The Butler constructor now accepts custom implementations of IButlerChatCollection. This allows developers to seamlessly swap the default sliding-window memory logic for custom, distributed persistence (e.g., Redis, SQL) while leaving the core orchestration intact.

TrenchCoat Engine Refactor: The default TrenchCoatChatCollection has been refactored to implement the new IButlerTrenchImplementation interface, which is required to support advanced tool-prompt steering and directives.

Context Window Age-Out: Introduced automatic state trimming. As new messages enter the running context window, the oldest standard messages are aged out and dropped to preserve token limits, without affecting critical System Prompts or injected Tool Directives. This does not effect the audit log.

Provider Metadata Hooks: Introduced a new lifecycle hook to handle Provider-Specific Tool Execution. This prevents state loss between turns by allowing providers to inject and persist highly specific metadata into the tool-call history.


The Bench (the tool collection):  Multithread Guard is made obsolete. It's always active now. This Effects the public AddTool routines, updating tool call 
budget/limits,  Removing tools, GetTool(), Exist Tool().  Notable this does *not* effect Async calling but *does* effect Sync based tool calling.

The Bench (the tool collection): New Feature. Supply your ApiKeyRateLimiter flavor thru Dependency Injection in the constructer. It also rejects null.

Tool Discoverer (reflection based tool scanner): Respects tool surface scope by default if assigned permssion. It also now has a dependancy on ButlerSDK.Tools.ToolSurface
Tool Discoverer (reflection based tool scanner): Default resource search term is now virtual as well as it's initizailtion routine. Enjoy the customization
Tool Discoverer (reflection based tool scanner): New resource: ToolSurfaceButlerTool_DiscovererResource. This one accepts a tool surface flag for allowed to be added to llm

🤖 Provider Updates

Google Gemini State Persistence: Fixed a critical state-loss issue where Gemini's internal "Thought Traces" were dropped during tool execution, which previously resulted in subsequent "Invalid Function" errors and HTTP error 400.

Google Gemini Pragmatic Shims: Added a conversation history validator. To prevent HTTP 400 errors from the Gemini API when history starts with a Model/System response, Butler now seamlessly injects a placeholder User message ("Hello") if the the chat log is empty (not counting system messages) or starts with a assistant role.

Reserved Keyword Notice: "GeminiModelThinking" is now a reserved provider-specific keyword used to route model thought-traces. Custom provider implementations should avoid utilizing this specific key in the ProviderSpecific dictionary to prevent collision if hot-swapping to Gemini. To be clear this this keyword is using in the dictionary<string,string> that houses provider specific strings.


Exception Propagation: Fixed a critical fault where mid-stream provider exceptions (such as API Key rejections or rate limits) were being silently swallowed. Butler now correctly surfaces unrecoverable provider errors directly to the calling application.

🐛 Bug Fixes

Core: Hardened the logic within ButlerMessageStitcher to robustly handle fragmented and malformed stream packets.

Tooling: Refactored state evaluation in ButlerTool_RestAPI_GetPublicHolidays to eliminate an infinite loop vulnerability caused by invalid JSON payloads.

ApiKeyRateLimiter (ButlerSDK's limiter for runaway LLM calls): fixed a case where if call budget is ok but not of inventory, budget was *still* subtracted instead of failing fully.

ApiKeyRateLimiter (ButlerSDK's limiter for runaway LLM calls): Each public api currently synchs via a private object for the class now. This should help prevent multh threading issues.


⚠️ Possible Risks & Migration Notes

Gemini Provider: The reserved keyword GeminiModelThinking is exclusively for Gemini Provider. Using it outside of Gemini Provider and later swapping to the provider may corrupt thought-trace state. Other providers can safely ignore this restriction.

Tool Memory Access: Tools accessing past context messages may encounter aged-out messages due to automatic trimming. BYOCC (Bring Your Own Chat Collection) implementations must handle this explicitly.

Multi-Version / NuGet Mismatch: Only v1.1.0-preview is officially supported. Older versions or mixed package versions may cause type or state inconsistencies. Developers are strongly advised to align package versions.

Async vs Sync Execution: If a tool defines both sync and async routines, async will be preferred. Sync calls may behave differently than prior versions — document accordingly.

Direct Internal Calls: Calling internal methods of ApiKeyRateLimiter or other core classes outside the public API is unsupported and may introduce threading or state issues.

Legacy / Untagged Tools: Untagged tools default to MaxPermissions. Ensure tool attributes are up-to-date to avoid unexpected permission failures.

The ToolSurfaceScope call is live on AddTool and when a tool is invoked by the LLM (ie ToolResolver).
